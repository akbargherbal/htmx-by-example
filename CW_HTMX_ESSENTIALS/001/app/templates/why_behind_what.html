The "Why" Behind the "What": HTMX Core Concepts

Why use `hx-get`?

The Problem: Normally, to fetch data and update a page, you need ~10 lines of JavaScript: write a `fetch()` call, handle the promise, parse the response, find the target DOM element, and set its `innerHTML`.

The HTMX Solution: You add one attribute: `hx-get="/your-url"`. HTMX handles the entire process. It's declarative, simple, and lives right in your HTML.

Why use `hx-post`?

The Problem: A standard form submission reloads the entire page. To prevent this, you'd write JS to intercept the `submit` event, manually collect form data, and send it via `fetch()`.

The HTMX Solution: `hx-post` on a form automatically prevents a full-page reload, gathers all input data, and sends it via an AJAX request. Zero custom JavaScript required.

Why use `hx-swap` and `hx-target`?

The Problem: After getting data, you need to decide *where* it goes and *how* to place it. Does it replace the content? The whole element? Go before or after it?

The HTMX Solution: `hx-target` lets you point to any element using a CSS selector. `hx-swap` gives you full control. `innerHTML` (the default) is great for content areas. `outerHTML` is powerful for replacing a component (like our soup) or removing it entirely (like our toast).

Why use `hx-trigger`?

The Problem: To get live updates, you'd typically write a `setInterval` function in JavaScript to repeatedly call an endpoint, which can be complex to manage.

The HTMX Solution: `hx-trigger="every 2s"` turns any element into a polling machine with a single, readable attribute. This is ideal for status monitors, dashboards, and live comment feeds.
